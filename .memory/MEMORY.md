# Repository Memory

## Stable Context
- **Issue‑driven工作流程**  
  - 所有工作皆以 GitHub Issue 為單位展開。  
  - 代理人（龍蝦）會每日掃描開放的 Issue，根據 Issue 內容產生摘要與待辦清單。  
  - 只有在 Issue 具備 **完整的需求說明、目標與驗收標準** 時，代理人才能進一步分配任務、標籤或制定執行計畫。

- **手動維護的長期記憶**  
  - `shared/manual.md` 用於保存 **穩定規則、長期決策、常見限制與 repo 習慣**，不會被自動覆寫。  
  - 代理人會參考此檔案以確保行為符合團隊共識。

- **代理人行為準則**  
  1. **資訊完整性優先**：若 Issue 缺少描述或目標，代理人僅記錄「資訊不足」並等待主人補充。  
  2. **不直接複製原始 Issue 文字**：摘要必須是蒸餾後的可讀資訊，保持工程可讀性。  
  3. **每日快照**：每一天產出一次 `daily/YYYY-MM-DD.json`，內容包括 Agent Activity、Cross‑Issue Themes、Decisions、Open Loops、Top Labels。  
  4. **限制條件**：  
     - 只檢視最近 30 天內、最多 100 個開放 Issue。  
     - 若無新 Issue 出現，僅重複「資訊缺口」主題時，仍須產出快照以提醒主人。

- **決策流程**  
  - 任何正式決策（功能範圍、優先順序、技術選型）必須在 Issue 中明確記錄，代理人才能將其納入長期記憶。  
  - 目前尚未有任何決策被記錄。
