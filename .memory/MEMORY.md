# Repository Memory

## Stable Context
- **Issue 建立規範**  
  - 必須提供完整的需求說明（背景、目標、驗收條件）。  
  - 建議使用統一的 Issue 模板，以避免「資訊缺口」的情況。  
  - 每個 Issue 應在建立時即指派負責人、加上適當的標籤（如 `needs‑info`、`in‑progress`、`blocked`），以提升可追蹤性。

- **溝通渠道統一**  
  - 任何補充說明、討論或決策應集中於已同步的平台（Telegram、Slack 等），並在 Issue 中留下連結或摘要。  
  - 龍蝦團隊會在每日快照中提醒缺失的外部資訊，確保資訊不會斷層。

- **決策流程**  
  - 只有在需求完整、目標明確且優先順序確定後，才會進行決策與任務分配。  
  - 目前尚未有任何正式決策，所有行動皆待主人提供關鍵資訊後才能啟動。

- **共享手動筆記** (`shared/manual.md`)  
  - 用於保存長期穩定規則、重要決策與常見限制。  
  - 任何自動化流程（如 compact‑memory）會讀取此檔案，但不會覆寫。

## Recent Themes
- **資訊缺口**：連續七天的快照皆顯示唯一的 Issue #1 只有標題，缺乏描述、目標與驗收條件，導致工作無法展開。  
- **需求澄清**：龍蝦們持續呼籲主人提供具體需求、背景說明以及相關即時訊息的連結。  
- **工作流啟動**：在缺乏需求資訊前，所有工作流（指派、標籤、優先排序）皆處於待命狀態。  
- **追蹤機制**：多次建議在 Issue 建立時即加入追蹤屬性（標籤、指派），以免未來重
