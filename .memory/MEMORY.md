# Repository Memory

## Stable Context
- **Issue 完整性是前提**  
  - 每個 GitHub Issue 必須至少包含：標題、問題描述、目標或需求、預期成果。缺少任一要素即視為「資訊不明」且無法啟動工作。  
- **需求先行、再分配**  
  - 在需求說明完整之前，不能指派負責人、加標籤或設定里程碑。需求確定後才可進行任務分配與優先排序。  
- **溝通渠道統一**  
  - 所有與 Issue 相關的討論（Telegram、Slack、Email 等）應在 Issue 內以引用或摘要方式呈現，避免資訊孤島。  
- **repo 習慣**  
  - 只使用 GitHub Issue / Comment 作為原始資料來源。  
  - `shared/manual.md` 為手動維護的長期記憶，**不會被自動覆寫**，僅供 `compact‑memory` 工作流讀取。  
  - 任何自動化產出的記憶（如 daily snapshots）必須 **去除原始文字**，只保留概念與結論。  
- **決策產出條件**  
  - 只有在需求、範圍與資源分配明確後，才能記錄正式決策與行動計畫。
