# Repository Memory

## Stable Context
- **Issue 建立慣例**  
  - 必須提供完整的需求說明（背景、目標、驗收條件）。  
  - 若有即時訊息（Telegram、Slack 等）作為補充，需在 Issue 中加入連結或摘要。  
  - 建議使用統一的 Issue 模板，以避免「資訊缺口」的情況。  
- **追蹤與指派**  
  - 每個 Issue 應在建立時即設定負責人（Assignee）與適當的標籤（Label），以提升可見性與後續追蹤效率。  
- **決策流程**  
  - 任何決策必須在需求說明完整且已獲得共識後方可記錄。  
  - 決策結果應同步更新至 Issue 狀態（如 `in‑progress`、`closed`）並在說明中說明原因。  
- **溝通渠道**  
  - 主要的需求與討論資訊來源為 GitHub Issue / Comment。  
  - 即時訊息平台僅作為補充，且必須在 Issue 中留下可追溯的引用。  

## Recent Themes
- **資訊缺口持續出現**  
  - 從 2026‑03‑22 起的唯一 Issue #1 連續多天皆缺乏描述、目標與討論，導致所有龍蝦無法展開工作。  
- **需求澄清需求**  
  - 每日快照均提醒主人提供具體需求、背景與預期成果。  
- **追蹤機制不足**  
  - 缺少標籤與指派，形成「等待回饋」的循環。  
- **模板與流程建議**  
  - 多次建議在建立 Issue 時使用包含 `title、背景、目標、驗收條件` 的模板，以減少資訊不明的情況。  

## Constraints
- **不可接受的 Issue 狀態**：  
  - 沒有任何描述或目標的 Issue 不能被視為可執行任務。  
- **必須提供的資訊**：  
  1. 問題背景與動機  
  2. 明確的目標或交付物  
  3. 相關限制或前置條件  
  4. 若有即時訊息補充，必須提供連結或摘要
